# V1 编辑模型

## 目标

这份文档定义 AI 如何真正“帮用户优化简历”，而不是只输出一堆解释。

核心思想是：

> 简历是一个结构化文档，AI 返回的是优化建议和可执行的文本改法，预览和 PDF 都由同一份文档渲染。

## 文档结构

简历至少要分成这些可编辑块：

- 基本信息
- 教育经历
- 工作经历
- 项目经历
- 专业技能
- 补充信息

每个块内部再拆成可定位节点，例如：

- 字段
- 段落
- 列表项
- 图片
- 标题

## 编辑操作

V1 至少支持三种操作：

### add

新增一段内容、一个列表项、一个字段或一个章节。

V1 中最重要的新增场景是：

> 用户输入一条新增经历，AI 自动判断应该放入工作经历、项目经历、专业技能还是补充信息，并生成插入后的简历文本。

### modify

修改已有内容，例如：

- 改写措辞
- 突出某个技术点
- 压缩长度
- 重新排序
- 改成更适合简历成品的表达

### delete

删除不需要的内容，例如：

- 重复或弱相关的列表项
- 过时信息
- 用户不想展示的信息

可选再支持：

- `move`
- `merge`
- `split`

## 新增经历操作

新增经历应该被建模为独立操作，而不是普通聊天回复。

推荐操作名：

- `insert_experience`

用户输入示例：

- “我最近做了一个 Redis 缓存优化，把接口平均响应时间从 800ms 降到 300ms。”
- “新增一个项目：AI 简历编辑器，前端 React，后端 Rust，支持 PDF 导出。”
- “我负责过 Docker 部署和 CI/CD 流程优化。”

AI 必须输出：

- 应该插入到哪个章节。
- 应该插入到哪个现有条目附近。
- 插入后的正文。
- 是否需要用户确认事实。
- 是否有缺失信息需要补充。

如果无法判断位置，AI 应优先让用户选择章节，而不是乱插。

## AI 输入

AI 可以接收：

- 当前简历结构化文档
- 用户修改指令
- 用户新增经历
- 之前的修改历史
- 用户已经提供并确认的简历内容

用户可以说：

- “把工作经历里和 Rust 无关的内容删掉。”
- “把项目经历改得更像专业简历里的项目描述。”
- “新增一条关于性能优化的经历。”
- “把这段压缩成两行。”
- “把这条新增经历放进简历里。”

## AI 输出

AI 的输出必须是结构化的，不只是自然语言。

推荐包含：

- `suggestions`
- `before`
- `after`
- `changeSummary`
- `insertedSection`
- `insertedPosition`
- `needsConfirmation`
- `blockedChanges`
- `riskNotes`

## 事实规则

AI 可以重写表达，但不能发明事实。

允许：

- 重新组织顺序
- 提炼用户已给出的经历
- 用更适合简历成品的语言表达同一事实

不允许：

- 编造公司、时间、职位
- 编造数字、指标、工具、结果
- 把模型推测误写成用户已经具备的事实

如果事实不足，优先：

- 提示用户确认
- 标记 blocked
- 不进入可导出状态

## 预览模型

预览不是另一份结果文本，而是同一份结构化文档的渲染视图。

因此：

- 预览改了，PDF 也必须改。
- 预览排版和 PDF 排版必须同源。
- 用户看到的就是最终稿长什么样。

## 确认模型

事实确认可以保留，但应作为辅助信息存在。

每个可疑或敏感的改动都应该能关联到：

- 用户原始输入
- 用户新增经历
- 用户手动确认

但主界面不要把确认信息做成审查主体。

## 状态

每个编辑项建议有这些状态：

- `draft`
- `needs_confirmation`
- `needs_user_input`
- `applied`
- `rejected`
- `blocked`

## 撤销和版本

至少要支持：

- 撤销最近一次 AI 操作
- 回到上一个稳定版本
- 保存当前草稿状态

不要求一开始就做完整历史版本库，但必须能让用户在试错时回头。
