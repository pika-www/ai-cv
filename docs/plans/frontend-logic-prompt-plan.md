# 前端逻辑 Prompt 计划

在接入前端状态、API 请求、表单、上传或用户流程之前，先使用这份提示词。

## 角色

你正在实现 AI CV Studio 的前端产品逻辑。界面必须响应快、表达诚实：不能暗示 AI 已经验证了用户没有提供过的事实。

## 核心流程顺序

按下面顺序实现前端流程：

1. 职业档案录入
2. 简历上传和文本解析交接
3. AI 追问
4. 目标岗位 JD 分析
5. 针对目标 JD 的简历内容生成
6. 导出

## V1 范围约束

当前 V1 面向自用和朋友试用，先完成“旧简历/经历 + 目标 JD -> 定制简历 bullet -> 来源追踪 -> 复制或导出”的核心闭环。

求职信、面试自我介绍、多版本管理、完整账号体系和公开发布能力属于后续范围。除非用户明确要求，不要在 V1 前端逻辑里提前铺这些主流程。

V1 使用分步骤流程，不使用三栏工作台作为主交互。步骤至少覆盖：开始与隐私说明、导入经历、确认职业档案、粘贴目标 JD、查看匹配与缺口、回答 AI 追问、生成简历内容、审查来源与风险、复制或导出。

## 状态模型

使用明确的状态名：

- `idle`：未开始
- `editing`：编辑中
- `uploading`：上传中
- `analyzing`：分析中
- `needs_evidence`：需要补充证据
- `ready_to_generate`：可以生成
- `generating`：生成中
- `generated`：已生成
- `failed`：失败

避免使用难以理解的布尔组合，例如 `isLoading && hasResult && !error`。

## API 边界

前端应该通过一个小的 API client 层调用后端。不要把原始 `fetch` 调用散落在各个组件里。

当前预期后端接口：

- `GET /health`
- `POST /api/profile/intake`
- `POST /api/jobs/match`
- `POST /api/materials/generate`

## 证据规则

展示 AI 输出时必须做到：

- 标明每条内容是否由用户职业档案支持。
- 标明哪些内容来自目标岗位 JD。
- 不支持的内容要标记为“无法生成”或“需要补充证据”，不要直接展示成成品。
- 不要默默编造数据、公司、职位、日期、工具或结果。

## 验收标准

- API 失败时显示清晰、可恢复的错误提示。
- API 请求前先完成前端校验。
- API 调用失败后，用户已经填写的内容不能丢失。
- 中文和英文使用同一套状态模型。
- 没有来源追踪摘要时，不展示生成材料成品。
