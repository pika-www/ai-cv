# 前端逻辑 Prompt 计划

在接入前端状态、API 请求、表单、上传或用户流程之前，先使用这份提示词。

## 角色

你正在实现 AI CV Studio 的前端产品逻辑。界面必须响应快、表达诚实：不能暗示 AI 已经验证了用户没有提供过的事实。

## 核心流程顺序

按下面顺序实现前端流程：

1. 粘贴简历文本或上传旧简历
2. 解析成可编辑草稿
3. 点击分析按钮
4. 右侧展示 AI 优化建议
5. 用户应用、忽略或编辑建议
6. 用户编辑简历
7. 用户输入结构化新增工作经历并自动插入
8. 预览照片和多页简历
9. 导出

## V1 范围约束

当前 V1 面向自用和朋友试用，先完成“粘贴/上传旧简历 -> AI 分析 -> 右侧建议 -> 左侧编辑 -> 新增经历自动插入 -> 导出 PDF”的核心闭环。

求职信、面试自我介绍、多版本管理、完整账号体系和公开发布能力属于后续范围。除非用户明确要求，不要在 V1 前端逻辑里提前铺这些主流程。

V1 使用单页编辑器作为主交互。页面至少覆盖：上传旧简历、粘贴简历文本、分析按钮、右侧建议、左侧编辑、新增经历输入、导出。

分析成功必须来自后端真实 AI provider 返回。前端 mock、固定规则和示例建议只能用于开发态，不能在验收或朋友试用时冒充 AI 分析结果。

## 状态模型

使用明确的状态名：

- `idle`：未开始
- `editing`：编辑中
- `uploading`：上传中
- `analyzing`：分析中
- `needs_user_input`：需要用户补充或确认
- `ready_to_export`：可以导出
- `exporting`：导出中
- `exported`：已导出
- `failed`：失败

避免使用难以理解的布尔组合，例如 `isLoading && hasResult && !error`。

## API 边界

前端应该通过一个小的 API client 层调用后端。不要把原始 `fetch` 调用散落在各个组件里。

当前预期后端接口：

- `GET /health`
- `POST /api/sessions`
- `POST /api/resumes/parse`
- `POST /api/resumes/analyze`
- `POST /api/resumes/insert-experience`
- `POST /api/exports`

## 事实规则

展示 AI 输出时必须做到：

- 标明每条建议对应简历里的哪段内容。
- 标明新增经历插入到了哪个章节。
- 不支持的改动要标记为“无法应用”或“需要用户确认”，不要直接展示成成品。
- 不要默默编造数据、公司、职位、日期、工具或结果。
- 建议的 `exampleRewrite` 必须是可直接应用到简历的成品句。
- 如果建议文本包含“请补充”“根据实际情况”“你确认过”“结果或范围”等提示语，前端应阻断一键应用。

## 新增工作经历表单

V1 新增工作经历使用结构化表单，至少包含：

- 公司名。
- 职位。
- 入职开始时间。
- 离职结束时间或“至今”。
- 主要项目名称。
- 项目介绍。
- 主要职责。
- 关键动作。
- 结果或产出。
- 自由描述。

前端向后端发送 `newWorkExperience` 对象。`newExperience` 纯文本字段只作为向后兼容，不作为新 UI 的主要数据结构。

## 预览与导出状态

前端必须把预览当作最终 PDF 的同源结果：

- 支持照片和基本信息。
- 内容超过一页时显示多页。
- 导出前检查所有页面都在导出范围内。
- 不允许预览多页但 PDF 只导出第一页。
- 不允许照片在预览正常但导出丢失或变形。

## 验收标准

- API 失败时显示清晰、可恢复的错误提示。
- API 请求前先完成前端校验。
- API 调用失败后，用户已经填写的内容不能丢失。
- 中文和英文使用同一套状态模型。
- 没有可用草稿时，不展示成品预览。
- 上传失败时必须允许用户改用粘贴文本继续。
- 分析请求在 provider 失败时显示可恢复错误，不能展示假建议。
- 用户能应用或忽略每条建议，应用后预览更新。
- 新增工作经历表单字段能进入后端请求并影响插入结果。
- 长简历能生成多页预览，导出不裁切内容。
