# V1 严格验收标准

## 使用方式

这份文档是 V1 的发布门槛。

如果任意 P0 条目不通过，不能认为 V1 可交给朋友试用。如果 P1 条目不通过，可以内部自用，但需要记录风险和修复计划。

## P0 发布阻断项

### 产品闭环

- 用户可以从开始页进入分步骤流程。
- 用户可以上传或粘贴经历。
- 用户可以确认或修正系统识别出的职业档案。
- 用户可以粘贴目标 JD。
- 系统可以展示 JD 要求、匹配证据和缺口。
- 系统可以提出与缺口相关的追问。
- 用户回答追问后，系统可以更新证据。
- 系统可以生成针对目标 JD 的简历 bullet。
- 用户可以看到每条 bullet 的来源。
- 用户可以复制或导出可用结果。

### 证据与生成

- 每条可导出的 bullet 必须有关联来源。
- 每条事实 claim 必须至少关联一个 EvidenceItem 或 JobRequirement。
- JD 只能证明岗位要求，不能单独证明用户拥有某项能力。
- 如果 claim 涉及公司、职位、日期、学历、工具、指标或结果，必须来自用户提供的证据。
- 没有证据支持的 claim 必须显示为 `needs_evidence` 或 `blocked`。
- `blocked` bullet 不能进入复制或导出结果。
- 未经用户确认的 `needs_confirmation` bullet 不能进入导出。
- 模型输出不是结构化 JSON 时，不能展示为成品。
- AI 生成失败时，用户已填写内容不能丢失。

### AI Key 与安全

- AI key 只能存在于后端环境变量。
- 前端源码、构建产物、浏览器 storage、网络响应中不能出现 AI key。
- `.env` 文件不能提交。
- 日志不能记录完整简历、完整 JD、手机号、邮箱、地址、API key 或原始 provider response。
- 后端错误不能泄露请求头、key、堆栈或 provider 原始错误全文。
- provider 调用必须有超时。
- provider 不可用时，前端必须显示可恢复错误。
- 小范围分享版本必须有访问控制，不能无保护公网开放。

### 上传与输入

- 不支持的文件类型必须被拒绝，并显示可理解错误。
- 文件超过大小限制必须被拒绝，并提示替代方式。
- 文件解析失败时，用户可以改用粘贴文本继续流程。
- 空简历、空 JD、过短 JD 不能进入依赖 AI 的后续步骤。
- 用户返回上一步修改输入后，下游分析和生成结果必须标记为过期。

### UI 与流程

- V1 必须是分步骤流程。
- 每一步必须显示当前步骤、下一步主操作和返回入口。
- 每一步必须有空状态、加载状态、成功状态和错误状态。
- 阻断条件必须显示具体原因。
- 390px 宽度下不能出现横向滚动。
- 中文和英文界面文案不能被按钮或容器裁切。
- 可交互元素必须有可见 focus 状态。
- 不能出现“保证通过 ATS”“保证拿面试”等承诺。

### 导出

- 导出前必须重新校验 bullet 状态。
- 复制结果不能包含 blocked 或 needs_evidence bullet。
- Markdown 导出不能包含 blocked 或 needs_evidence bullet。
- PDF 导出不能包含 blocked 或 needs_evidence bullet。
- 导出失败时，用户已编辑内容不能丢失。
- 导出文件名应包含日期和目标职位线索。

## P1 质量要求

### 产品体验

- 新用户不需要开发者解释，也能完成核心流程。
- 用户能在 10 到 20 分钟内完成一次“旧简历 + JD -> 定制简历 bullet”。
- 用户能理解为什么系统要追问某个问题。
- 用户能区分“已支持”“需要确认”“证据不足”“已阻断”。
- 朋友试用时，常见错误能通过界面自行恢复。

### AI 输出质量

- Bullet 应具体、简洁、贴合目标 JD。
- 中文输出不能像英文直译。
- 英文输出不能像中文直译。
- 没有量化指标时，AI 应生成定性表达，而不是编造数字。
- 用户提供估算数字时，应标记为用户估算或需要确认。
- ATS 建议只能围绕清晰度、关键词覆盖和可解析性，不能制造通过率承诺。

### 数据关联质量

- 每个 FollowUpQuestion 必须关联一个 JobRequirement 或 EvidenceItem。
- 每个 FollowUpAnswer 产生的新事实必须转成 EvidenceItem。
- 每个 GeneratedBullet 必须拆成 Claim。
- 每个 Claim 必须有 SourceReference。
- MaterialSet 必须记录 prompt version、provider 和 model name。

### 隐私与数据处理

- 上传原文件不应长期保存，除非用户明确选择。
- 结构化证据和原始文件应分开处理。
- 用户应能看到基础数据处理说明。
- 分享给朋友前，应准备一段简单隐私说明。

### 性能

- 普通文本粘贴后，页面应在 1 秒内可继续交互。
- AI 请求期间，界面必须显示进度或等待状态。
- 单次 AI 请求超时后必须返回错误状态。
- 大输入不能导致前端页面卡死。

## P2 可后置但建议尽快补齐

- DOCX 导出。
- 简历版本历史。
- 多个 JD 对比。
- 用户可删除当前 session 数据。
- 更细的 prompt 版本变更记录。
- 管理员查看请求量和预算消耗。

## 手动验收用例

### Case 1: 正常流程

输入：

- 一份包含 2 段工作经历的旧简历。
- 一个具体目标 JD。
- 2 到 3 个追问回答。

必须通过：

- 生成 4 到 8 条可用 bullet。
- 每条 bullet 有来源。
- 至少一个 JD requirement 有匹配证据。
- 至少一个缺口被转成追问。
- 结果可以复制和导出。

### Case 2: 证据不足

输入：

- 简历只写“参与项目开发”。
- JD 要求“主导高并发架构优化，提升系统吞吐量”。

必须通过：

- 系统不能直接生成“主导高并发架构优化”。
- 系统必须追问职责、规模、指标、工具或结果。
- 未回答前，相关 bullet 必须是 blocked 或 needs_evidence。

### Case 3: 用户提供估算数字

输入：

- 用户回答“响应时间大概从 900ms 降到 300ms，具体监控没保存”。

必须通过：

- 生成内容可以使用该数字，但必须标记需要确认或用户估算。
- 来源必须指向该用户回答。
- 导出前必须要求用户确认。

### Case 4: AI 编造检测

输入：

- 用户简历没有 Kubernetes。
- JD 要求 Kubernetes。

必须通过：

- 系统不能生成“使用 Kubernetes 部署服务”。
- 系统可以提示这是缺口。
- 系统可以追问用户是否有相关经验。

### Case 5: Provider 失败

输入：

- 正常简历和 JD。
- 模拟 AI provider timeout 或 invalid key。

必须通过：

- 页面显示清晰错误。
- 不泄露 key 或 provider 原始响应。
- 用户输入不丢失。
- 用户可以重试。

### Case 6: 导出阻断

输入：

- 生成结果中包含一条 needs_evidence bullet。

必须通过：

- 该 bullet 不能被复制到最终结果。
- Markdown 和 PDF 导出都必须排除或阻断。
- 页面解释为什么不能导出。

### Case 7: 移动端

环境：

- 390px 宽度。

必须通过：

- 无横向滚动。
- 步骤导航可见。
- 长文本不会遮挡按钮。
- 来源追踪可打开查看。

## 发布前检查清单

- P0 全部通过。
- 至少跑通 Case 1 到 Case 7。
- 使用一份真实简历样例完成端到端测试。
- 使用一份很差的简历样例完成证据不足测试。
- 检查前端构建产物中没有 AI key。
- 检查日志中没有完整简历和完整 JD。
- 确认分享入口有访问控制。
- 确认 README 或使用说明不会引导用户提交真实 key 到前端。

## V1 完成定义

V1 完成不是“页面都能打开”，而是：

> 一个朋友拿到链接后，可以在不理解技术细节的情况下，基于自己的真实经历和一个目标 JD，生成一组可追溯、可复制、没有明显编造事实的简历 bullet。
