# V1 严格验收标准

## 发布原则

如果下面的 P0 条目有任何一条不过，不能把 V1 给朋友试用。

如果 P1 条目不过，可以内部自用，但不算完成。

## P0 发布阻断项

### 核心闭环

- 用户能粘贴简历文本。
- 用户能上传旧简历文件。
- 上传失败时，用户能改用粘贴文本继续。
- 用户能点击分析按钮。
- 系统必须调用真实 AI provider 分析当前简历。
- 系统能在右侧返回可理解、可操作的 AI 优化建议。
- 用户能逐条应用、忽略或手动编辑建议。
- 用户能在左侧直接编辑简历。
- 用户能输入结构化新增工作经历。
- AI 能把新增经历放入简历合适位置。
- 用户能重复分析和编辑。
- 用户能导出 PDF。
- 导出的 PDF 和预览在版式上保持一致。
- 长简历能自然导出多页 PDF。

### AI 真实性

- V1 验收环境必须使用真实 AI provider 返回优化建议。
- 本地规则、mock 数据、固定模板建议不能作为验收通过依据。
- provider 未配置、无效、超时或 rate limit 时，必须显示可恢复错误。
- provider 失败时不能静默返回假建议。
- 分析结果必须来自当前草稿，不能复用旧草稿或示例简历的建议。
- 前端不能直接调用 AI provider。
- 用户必须可以选择使用默认模型 key，或输入自己的模型 key。
- 没有默认 key 且用户未配置 key 时，系统必须引导用户配置 key，不能假装分析成功。

### 编辑真实性

- AI 不能编造公司、时间、职位、学历、工具、指标或结果。
- 没有证据的内容不能直接进入可导出结果。
- 需要确认的内容必须明确标记。
- 被阻断的内容不能混进导出结果。
- 用户删掉的内容不能在下一轮无提示地回来。
- 新增经历只能基于用户输入生成，不能扩写成用户没有提供过的事实。
- `exampleRewrite` 和 `insertedText` 必须是可以直接进入简历的成品句。
- 可应用文本中不能出现“请补充”“根据实际情况”“你确认过”“结果或范围”等提示语。
- 信息不足时必须标记 `needsUserInput`、提出问题或阻断应用，不能把提示语写进成品简历。

### 简历成品与分页

- 简历预览必须像真实简历页面，而不是 Markdown 或纯文本 dump。
- 简历必须支持照片或头像展示。
- 简历必须支持姓名、电话、邮箱、城市、求职意向等基本信息。
- 简历必须支持教育、工作、项目、技能和补充信息等主要章节。
- 内容超过一页时，必须生成第 2 页、第 3 页等页面。
- 多页之间字体、页边距、章节样式和照片处理必须一致。
- 不能为了塞进一页而把字号压到不可读。
- 不能裁切、隐藏、重叠或导出丢失长内容。
- 预览有几页，PDF 就必须完整导出几页。

### 新增工作经历

- 新增工作经历表单必须包含公司名。
- 新增工作经历表单必须包含职位。
- 新增工作经历表单必须包含入职开始时间。
- 新增工作经历表单必须包含离职结束时间或“至今”。
- 新增工作经历表单必须包含主要项目名称。
- 新增工作经历表单必须包含项目介绍。
- 新增工作经历表单必须包含主要职责、关键动作和结果。
- AI 插入时必须基于这些字段生成文本。
- 缺失字段必须提示用户补充或标记待确认。
- AI 不能补写用户没有提供的公司、职位、时间、项目结果或薪资信息。

### AI Key 与安全

- 产品默认 AI key 只能在后端环境变量里。
- 用户自带 AI key 必须通过模型设置入口提交，并由后端代调 provider。
- 前端不能直接调用 AI provider。
- 前端源码、构建产物、URL、错误消息、导出文件里都不能出现完整 AI key。
- API 响应只能返回脱敏 key，例如 `sk-...abcd`。
- 用户必须可以清除当前会话 key。
- 用户 key 默认不能长期保存到服务器。
- 如果提供“记住 key”，必须默认关闭，并明确说明只保存在当前浏览器本地。
- `.env` 文件不能提交。
- 日志不能记录完整简历、手机号、邮箱、地址、API key 或原始 provider response。
- provider 调用必须有超时。
- provider 失败时必须显示可恢复错误。
- 朋友试用版本必须至少有简单访问控制，不能无保护公网开放。

### UI 与体验

- 主界面必须是简历编辑器，不是纯文本审计报告。
- 右侧建议不能抢走主视图。
- 上传入口、粘贴入口、新增经历入口都必须容易找到。
- 模型 key 设置入口必须容易找到。
- 多页预览和导出入口必须容易找到。
- 390px 宽度不能出现横向滚动。
- 中文长文案和英文长文案都不能裁切。
- 可交互元素必须有可见 focus 状态。
- 用户返回上一步时，已填内容不能丢。

### 导出

- 导出前必须重新校验当前草稿状态。
- blocked 或未确认的内容不能进入最终 PDF。
- 导出失败时，当前编辑内容不能丢失。
- PDF 文件名应包含日期和简历名称线索。
- PDF 必须包含当前预览中的照片、基本信息和所有页面。

## P1 质量要求

### 可用性

- 新用户不需要开发者解释，也能完成一次完整编辑。
- 用户能看懂 AI 为什么做了某个改动。
- 用户能在 10 到 20 分钟内把一份现有简历改到可导出状态。

### 预览一致性

- 预览和 PDF 使用同一模板。
- 页面分页、字体、间距、图片和分栏在预览和导出中保持一致。
- 不允许导出后出现明显排版漂移。

### AI 输出质量

- 修改结果要自然，不像机器拼接。
- 中英文文案不能直译腔。
- 没有数字时不要硬编数字。
- 用户提供估算时要标记为估算或待确认。
- 建议必须具体到可执行修改，不能只说“建议优化表达”。
- 例如原文只有“会 Excel”时，AI 应根据上下文给出具体可用表达；上下文不足时应追问，而不是写“补充你确认过的结果或范围”。

### 编辑模型

- 每个编辑项都有稳定的 ID。
- 每轮 AI 建议都能被回滚。
- 至少支持一个稳定草稿版本。
- 建议能映射到具体文本区域。

## P2 可后置但建议尽快补齐

- DOCX 导出。
- 简历版本历史面板。
- 目标岗位/JD 对比。
- 更精细的模板库。
- 用户删除当前会话数据。
- 细粒度操作日志。

## 手动验收用例

### Case 1: 导入并预览

输入：

- 一份真实旧简历。

必须通过：

- 页面立即出现可编辑文本。
- 用户能看见建议区和编辑区。
- 用户能看见接近 PDF 的简历预览。

### Case 1B: 上传旧简历

输入：

- 一份 `.pdf`、`.docx` 或 `.txt` 旧简历。

必须通过：

- 系统能解析出可编辑文本，或在解析失败时提供粘贴文本替代路径。
- 用户可以点击分析按钮。
- 后端真实请求 AI provider。
- 右侧能展示 AI 返回的优化建议。

### Case 1C: 用户配置模型 Key

输入：

- 后端没有配置默认 AI key。
- 用户打开模型设置，填写 provider、base URL、model 和 API key。

必须通过：

- 用户能保存当前会话 key。
- 页面只展示脱敏 key。
- 点击分析后，后端使用用户 key 调用 provider。
- key 不出现在 URL、错误消息、导出文件或日志中。
- 用户能清除当前 key。

### Case 1D: 模型 Key 未配置

输入：

- 后端没有默认 key。
- 用户也没有配置自己的 key。
- 用户点击分析。

必须通过：

- 页面提示需要配置模型 key。
- 不返回 mock 建议。
- 用户已输入或上传的简历内容不丢失。

### Case 2: add / modify / delete

输入：

- “把这段改得更简洁。”
- “帮我突出 Rust 后端经历。”
- “删除重复内容。”

必须通过：

- AI 输出具体建议。
- 用户能手动应用或忽略。
- 预览随之更新。
- 应用后的文本不能包含占位提示语。

### Case 2C: Excel 弱表述优化

输入：

- 原简历只有一句：“会 Excel”。
- 简历上下文显示用户做过销售数据、客户线索或业务报表。

必须通过：

- AI 建议把 Excel 能力细化到具体使用场景，例如数据清洗、透视表、报表维护或业务分析。
- `exampleRewrite` 是可以直接放进简历的句子。
- 如果上下文不足，AI 应提出问题，而不是生成带“请补充”的可应用文本。

### Case 2B: 新增经历自动插入

输入：

- 当前简历已有工作经历和项目经历。
- 用户在结构化表单填写公司名、职位、起止时间、项目名、项目介绍、职责、动作和结果。

必须通过：

- AI 判断应该放入工作经历或项目经历。
- 系统展示插入后的简历内容。
- 用户可以接受、编辑或撤销。
- 插入内容不能编造额外公司、时间、工具或结果。
- 插入后的文本不能包含“请补充”等提示语。

### Case 2D: 新增经历缺字段

输入：

- 用户只写“做了一个 Redis 优化项目”，没有公司、职位、时间或结果。

必须通过：

- 系统提示缺少关键字段，或让用户选择章节并标记待确认。
- AI 不能编造公司、职位、时间、性能指标或项目结果。
- 未确认内容不能进入最终 PDF。

### Case 3: 证据不足

输入：

- 用户没有提供某项指标。
- 但建议想写进简历。

必须通过：

- 系统阻断该内容或提示用户确认。
- 不能直接进入最终 PDF。

### Case 4: 导出一致性

输入：

- 完成一轮编辑后导出 PDF。

必须通过：

- 导出文件与预览一致。
- 不出现重新排版导致的明显错位。
- 照片、基本信息和章节内容完整保留。

### Case 4B: 多页导出

输入：

- 一份内容明显超过一页的简历，包含照片、基本信息、教育、两段工作经历和多个项目。

必须通过：

- 预览显示多页。
- PDF 导出页数与预览一致。
- 第 2 页及后续页面内容不裁切、不重叠、不丢失。
- 多页样式一致。

### Case 5: AI 失败

输入：

- 模拟 provider timeout 或无效 key。

必须通过：

- 页面显示可恢复错误。
- 用户已输入内容不丢失。
- 用户可以重试。

### Case 6: 移动端

环境：

- 390px 宽度。

必须通过：

- 无横向滚动。
- 预览可查看。
- AI 指令可输入。
- 导出入口可达。
